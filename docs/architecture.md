# YouTube + Twitch 实时直播 AI 翻译软件 — 架构与实施计划 v2

## 0. 重大架构变更说明

**核心思路反转：**
- **v1 思路**：yt-dlp/streamlink 直连直播 URL 提取音频流 → 翻译
- **v2 思路（当前）**：系统音频捕获为默认模式，直播 URL 直连降级为实验模式

**变更理由：**
1. 系统音频捕获 = 通用方案，无需适配每家直播平台，用户正常用浏览器看就行
2. 平台直连方案依赖 yt-dlp/streamlink，随时可能被平台反爬机制封堵
3. 系统音频捕获是用户感知最简单的路径："打开直播 → 打开翻译软件 → 完成"

## 3. 六大开发阶段

---

### 阶段一：基础设施与安装体验 (第 1-3 周)

**目标：** 用户能成功安装、启动应用、完成首次设置向导、看到 UI 框架。

#### 任务清单

| # | 任务 | 描述 | 优先级 |
|---|------|------|--------|
| 1.1 | 项目脚手架搭建 | Electron 31+ + React 18 + TypeScript + Vite + TailwindCSS，配置 electron-builder 打包 | P0 |
| 1.2 | Python 后端嵌入方案 | 调研并确定：PyInstaller 打包 vs embeddable Python vs 要求用户安装 Python。推荐 python-build-standalone 嵌入方案 | P0 |
| 1.3 | 首次安装向导 (SetupWizard) | 4 步向导：① 欢迎页 ② 硬件检测 + STT 模型推荐 ③ 模型下载（进度条 + 断点续传）④ 音频权限引导 + 完成 | P0 |
| 1.4 | 模型下载管理器 | 下载队列、断点续传、校验 hash、显示进度/速度/预计剩余时间、失败重试。模型：silero-vad.onnx (~2MB) + faster-whisper 模型 (100MB-3GB) | P0 |
| 1.5 | 硬件自动检测 | 检测：CPU 核心数/内存、GPU 型号/VRAM、CUDA 版本、磁盘空间。输出：推荐 STT 模型大小 + 预计延迟 | P0 |
| 1.6 | 应用更新框架 | electron-updater 集成，支持增量更新、静默下载、重启安装、更新日志展示 | P0 |
| 1.7 | 基础 UI 框架 | 主窗口布局（左侧导航 + 右侧内容区）、暗色/亮色主题切换、多语言 UI 框架 (i18n)、响应式适配 | P0 |
| 1.8 | 日志与诊断系统 | 结构化日志（客户端 + Python 端），日志级别可配，一键导出诊断包（日志+配置+系统信息） | P1 |

#### 测试要点
- 在 Windows 10/11、macOS 14+ 上完成安装向导全流程
- 模拟模型下载中断 → 断点续传恢复
- 模拟磁盘空间不足 → 友好提示
- 模拟无 GPU 环境 → 正确推荐 CPU 模型
- electron-updater 增量更新验证

#### 验收标准
- [ ] 全新机器上双击安装包 → 5 分钟内完成安装 + 模型下载 + 进入主界面
- [ ] 硬件检测准确率 > 95%（GPU 型号、VRAM、CUDA 版本）
- [ ] 模型下载支持断点续传，中断 3 次仍能完成
- [ ] 更新包增量 < 50MB（全量 < 300MB）
- [ ] Windows 7+ / macOS 12+ 均能正常启动

#### 风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| python-build-standalone 体积过大 | 安装包 > 500MB | 按需下载 Python runtime，不在安装包中捆绑 |
| 模型下载耗时长（3GB） | 用户流失 | 默认 tiny/small 模型，大模型可选下载；显示实用进度信息 |
| macOS 公证/签名失败 | 无法分发 | CI 中提前验证签名流程，预留公证缓冲时间 |
| 硬件检测在某些 GPU 上失败 | 推荐错误模型 | 提供手动选择模型大小的入口，覆盖主流 GPU 数据库 |

### 阶段二：系统音频捕获 + 核心 Pipeline (第 4-7 周)

**目标：** 捕获系统音频 → VAD → STT → 翻译 → 屏幕上显示翻译结果。这是系统最核心的价值链路。

#### 任务清单

| # | 任务 | 描述 | 优先级 |
|---|------|------|--------|
| 2.1 | Windows 音频回路捕获 | 使用 WASAPI Loopback (node-record-lpcm16 或自定义 native addon)，捕获系统音频输出 | P0 |
| 2.2 | macOS 音频回路捕获 | 使用 CoreAudio / AVAudioEngine，或 BlackHole/Soundflower 虚拟设备方案 + 权限引导 | P0 |
| 2.3 | 统一音源抽象层 (UnifiedAudioCapture) | 定义 `AudioSource` 接口：`start() / stop() / read() / getDevices()`，实现系统音频 + 麦克风 + 直播直连三种适配器；支持运行时热切换音源 | P0 |
| 2.4 | 音频规格归一化 (AudioNormalizer) | 所有音源统一输出为 16kHz / mono / 16-bit PCM，使用 FFmpeg 或 Web Audio API 重采样 | P0 |
| 2.5 | RingBuffer 环形缓冲 | 线程安全、可配置容量 (5-15s)、背压保护（>80% 丢弃最旧数据）、水位监控 | P0 |
| 2.6 | silero-vad 集成 | 32ms 帧级别语音概率检测，可调阈值 (0.3-0.7)，min_speech_duration / min_silence_duration 可配 | P0 |
| 2.7 | faster-whisper 集成 | 模型加载、预热、转写（含词级时间戳）、多模型大小支持、GPU/CPU 自动切换 | P0 |
| 2.8 | SentenceSegmenter（多语言标点感知） | 中/英/日/韩标点感知：中文 `。！？`、英文 `.?!`、日文 `。！？`、韩文 `.?!`；静音间隙 > 700ms 强制分段；最大句长 200 字符；处理省略号 `...` 和引号嵌套 | P0 |
| 2.9 | Pipeline Orchestrator | 串联 AudioNormalizer → RingBuffer → VAD → STT → SentenceSegmenter → TranslationQueue，asyncio 协程流水线，可暂停/恢复/停止 | P0 |
| 2.10 | LibreTranslate 本地翻译集成 | 管理 LibreTranslate 服务器生命周期（首次启动自动下载模型 ~1-2GB），REST API 调用，健康检查 | P0 |
| 2.11 | Python ↔ Electron WebSocket 通信 | FastAPI WebSocket 端点，消息协议定义，心跳 (5s)，重连逻辑（指数退避 1s→2s→4s→...→max 30s），端口自动分配 | P0 |

#### 测试要点
- 在 YouTube Live / Twitch 上播放测试直播 → 验证系统音频成功捕获
- 在 Bilibili 直播 / 抖音直播上播放 → 验证跨平台兼容（系统音频方案天然支持）
- 播放纯音乐 → VAD 有效过滤，不触发 STT
- 快速切换说话人 → 分段正确
- 中英混合直播 → 句子分割正确
- STT 模型在 CPU 上也能达到实时（RTF < 0.5 即 1 秒音频处理 < 0.5 秒）
- 连续运行 2 小时无内存泄漏

#### 验收标准
- [ ] 音频捕获延迟（从系统出声到 Pipeline 收到 PCM）< 100ms
- [ ] VAD 语音检测准确率 > 90%（与人工标注对比）
- [ ] STT 在 small 模型 + GPU 上的实时率 RTF < 0.3（即 1 秒音频处理 < 300ms）
- [ ] 端到端延迟（说话 → 翻译显示）< 5 秒
- [ ] 连续运行 4 小时：内存增长 < 200MB，无崩溃
- [ ] macOS 权限引导清晰，用户一次授权即可

#### 风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| macOS 15+ 收紧音频权限 | 无法捕获系统音频 | 准备 BlackHole 虚拟设备方案作为备选；引导用户安装 |
| Windows WASAPI 独占模式冲突 | 其他应用无法播放声音 | 使用共享模式 loopback，检测冲突并提示 |
| faster-whisper 在低端 CPU 上太慢 | 延迟 > 10 秒 | 自动降级到 tiny 模型或 whisper.cpp |
| LibreTranslate 首次模型下载失败 | 翻译不可用 | 预置 Argos Translate 作为最小可用翻译回退 |
| 多平台音频 API 差异大 | 开发周期拉长 | 先只做 Windows（用户最多），macOS 放 Phase 2.5 |

### 阶段三：延迟控制 + 字幕队列 + 中文错误恢复 (第 8-11 周)

**目标：** 用户可精确控制翻译延迟、字幕显示平滑流畅、中文翻译结果经过纠错优化。

#### 任务清单

| # | 任务 | 描述 | 优先级 |
|---|------|------|--------|
| 3.1 | 延迟控制系统 | 三级延迟：① 音频缓冲延迟（RingBuffer 大小，1-15s）② 分段延迟（min_silence_duration，200ms-1s）③ 显示延迟（SubtitleQueue 延迟，0-5s）。总延迟 = ①+②+③，控制在 3-15 秒可调。用户拖动一个总延迟滑块，系统自动分配三级参数 | P0 |
| 3.2 | SubtitleQueue（字幕队列管理器） | 按 `start_ms` 排序的优先队列，去重（相同文本 5s 内不重复显示），显示节奏控制（最小显示时长 1.5s，最大 8s），平滑过渡（淡入淡出 200ms），突发处理（积压 > 10 条时合并显示 "..." 省略，追赶实时） | P0 |
| 3.3 | 中文错误恢复 (ChineseErrorRecovery) | 四层纠错：① 同音字纠错（"在"→"再"，"的"→"得"，基于拼音混淆矩阵）② 上下文连贯修复（相邻句语义冲突检测，用轻量 BERT 做）③ 专有名词保护（人名/地名/品牌名不纠错，维护白名单）④ Whisper 已知错误模式修复（中英混合、数字、重复词） | P0 |
| 3.4 | 翻译结果缓存 | LRU + TTL 双层缓存：短语级缓存 (TTL 30s, max 2000 条) + 句子级缓存 (TTL 60s, max 500 条)，hash key = `(source_lang, target_lang, text)` | P1 |
| 3.5 | 翻译回退链 | 用户配置优先级列表：默认 [LibreTranslate → Argos Translate → 显示原文]，可添加 [DeepL API → GPT-4o-mini]。每个后端有独立超时 (3s) 和熔断器 (3 次失败 → 冷却 30s) | P1 |
| 3.6 | 字幕样式系统 | 预设主题：① 经典（黑底白字）② 半透明（半透明黑底白字圆角）③ 极简（纯白字黑描边无底）④ 电竞（霓虹绿黑底等宽字体）。自定义：字号(12-48px)、字体、颜色、背景色/透明度、位置(上/中/下 + 左/中/右 9 宫格)、最大行数(1-4)、滚动方向 | P1 |

#### 测试要点
- 延迟滑块从 3s 拖到 15s → 测量实际端到端延迟误差 < ±500ms
- 高速说话（4 字/秒中文）→ SubtitleQueue 正确处理积压
- 同音字纠错准确率：输入 100 句含常见同音错别字的文本 → 纠错后准确率 > 85%
- 上下文连贯：检测 "我今天去了超市。所以买了苹果和香蕉。" 中无错误；检测 "我今天去了超市。所以卖了苹果和香蕉。" 中 "卖" 应为 "买"

- [ ] SubtitleQueue 在 2x 实时语速下不丢句、不卡顿

|------|------|----------|

## 5. 核心架构决策 (更新)

### 决策 1: 系统音频捕获为默认模式（★ 关键变更）

**选择:** 90% 的用户场景使用系统音频回路捕获，直播 URL 直连降级为 "实验模式"。

**理由:**
- 系统音频捕获是通用方案：YouTube、Twitch、Bilibili、抖音、TikTok 等任何平台都支持
- 用户无需学习任何额外操作：正常打开浏览器看直播 → 打开本软件 → 完成
- 无需维护 yt-dlp/streamlink 适配器应对平台反爬更新
- 音频权限引导是一次性的，之后无感

**代价:** 需要处理各操作系统的音频 API 差异 (WASAPI / CoreAudio / PulseAudio)，权限引导在不同 OS 上有不同 UX。

### 决策 2: 中文错误恢复作为独立模块

**选择:** 在 STT 和翻译之间插入 ChineseErrorRecovery 模块，专门处理中文 STT 常见错误。

**理由:**
- Whisper 对中文的同音字/多音字错误有固定模式，可以针对性修复
- 中国用户是首要目标用户群体，中文体验是核心差异化竞争力
- 独立模块便于持续迭代优化，不影响其他语言的处理路径
- 上下文连贯性检查能显著提升翻译质量（输入质量决定输出质量）

### 决策 3: SubtitleQueue 引入显示缓冲层

**选择:** 在 WebSocket broadcast 之前插入 SubtitleQueue，管理字幕显示节奏。

**理由:**
- STT + 翻译的输出是不均匀的（有时爆发、有时空闲），直接显示会导致字幕跳跃
- SubtitleQueue 提供平滑的显示体验：最小显示时长保证可读性，积压时智能合并
- 延迟控制在这里有第三个可调节点（显示延迟），与音频缓冲延迟和分段延迟协同

### 决策 4: 弱网自适应而非仅代理支持

**选择:** 不止是代理配置，而是主动检测网络质量并自适应调整 Pipeline 策略。

**理由:**
- 直播观看场景中网络波动是常态（特别是移动热点用户）
- 自适应策略避免用户在信号差时手动调参
- 降级路径：大模型→小模型→仅 STT（不翻译）→ 显示网络错误
- 网络恢复后自动恢复全功能，无需用户干预
